# План монетизации Snappy

## Цель документа

Этот документ фиксирует простую и стабильную модель монетизации продукта:

- пользователь пополняет баланс в продукте;
- раз в месяц с баланса списывается фиксированная подписка;
- каждый запрос к AI списывается по фактической себестоимости AiTunnel с единым коэффициентом `K`;
- тарифы для пользователя не плавающие (коэффициент меняется редко или не меняется вовсе).

Модель ориентирована на текущий статус владельца как **самозанятый (НПД)**.

## Продуктовая модель (без плавающих тарифов)

### 1) Баланс пользователя

- Пользователь пополняет внутренний баланс через YooKassa.
- Баланс используется как кошелек для:
  - ежемесячной подписки;
  - списаний за AI-запросы.

### 2) Подписка

- Раз в месяц автоматически списывается фиксированная сумма подписки `S`.
- Подписка оплачивает доступ к продукту, интерфейсу, функциям и сервисной части.
- Подписка не означает безлимитное потребление AI (если это не указано отдельно в оферте).

### 3) Списание за AI-запрос

- Для каждого запроса используем фактическую стоимость от AiTunnel: `cost_rub`.
- Пользовательское списание считается по формуле:

`charge_rub = round_up(cost_rub * K, 2)`

Рекомендуется также минимальное списание:

`charge_rub = max(min_charge, round_up(cost_rub * K, 2))`

Где:

- `K` — фиксированный коэффициент безубыточности и маржи;
- `min_charge` — минимальное списание за запрос (например `0.05` руб), чтобы не терять деньги на микро-стоимостях и
  округлениях.

## Почему единый фиксированный K подходит

- В коде уже есть фактическая себестоимость в рублях (`cost_rub`) из AiTunnel.
- Основной риск изменения цен моделей уже встроен в саму себестоимость.
- Пользователи получают прозрачное правило: одна формула для всех запросов.
- Продукт не превращается в динамический прайс с частыми изменениями.

## Расчет K (как выбрать один раз)

### Базовая логика

`K` должен покрывать:

- НПД;
- комиссию YooKassa;
- возвраты/списания/операционные потери;
- целевую маржу бизнеса.

Упрощенная формула нижней границы:

`K_min = 1 / (1 - tax_rate - acquiring_rate - loss_rate - target_margin_rate)`

Где:

- `tax_rate` — эффективная налоговая доля (НПД);
- `acquiring_rate` — средняя комиссия приема платежей;
- `loss_rate` — возвраты, округления, мошенничество, скидки и прочие потери;
- `target_margin_rate` — желаемая маржа именно на AI-списаниях.

### Пример

Если:

- `tax_rate = 0.04` (4%);
- `acquiring_rate = 0.03` (3%);
- `loss_rate = 0.02` (2%);
- `target_margin_rate = 0.08` (8%);

то:

`K_min = 1 / (1 - 0.04 - 0.03 - 0.02 - 0.08) = 1 / 0.83 = 1.2048`

Практически: взять `K = 1.21` или `K = 1.22`.

## Рекомендуемая финансовая конфигурация

- Фиксированный публичный `K` (например `1.20`–`1.25`) с запасом.
- Подписка `S` покрывает:
  - фиксированные расходы (инфраструктура, поддержка, разработка);
  - прибыль продукта;
  - волатильность поведения пользователей.
- AI-расходы покрываются отдельно через `cost_rub * K`.

Такая комбинация дает:

- предсказуемость для пользователя;
- контроль unit-экономики;
- минимальный риск работать в минус по AI-вызовам.

## Налоговый контур для самозанятого (НПД)

Важно: это операционные рекомендации, а не индивидуальная налоговая консультация.

### Основной принцип

- НПД считается по факту денежных поступлений от клиента.
- Внутренние проводки в балансе пользователя (списание подписки, списание запроса) сами по себе не создают новый
  денежный поток.
- Расходы на AiTunnel при НПД не уменьшают налоговую базу.

### Практический учет

- На каждое входящее пополнение от пользователя формировать чек НПД по правилам сервиса/интеграции.
- При возврате пользователю оформлять корректировку/возврат чека в установленном порядке.
- Держать связку `payment_id -> user_id -> сумма -> чек -> статус`.

## Юридическая прозрачность (оферта и тарифы)

В публичных документах (оферта/тарифы/FAQ) зафиксировать:

- Подписка списывается ежемесячно с баланса.
- AI-вызовы оплачиваются отдельно по формуле:
  - фактическая стоимость провайдера в рублях;
  - умноженная на фиксированный коэффициент `K`.
- Минимальное списание за запрос (если используется).
- Порядок возвратов и корректировок.

## Операционный регламент (ежемесячно)

Даже при фиксированном `K` нужно внутренне проверять экономику:

1. Посчитать за месяц:
   - входящие платежи;
   - сумму AI-списаний у пользователей;
   - фактические расходы AiTunnel;
   - комиссии YooKassa;
   - налог НПД;
   - возвраты/потери.
2. Проверить маржу:
   - по подписке;
   - по AI-списаниям;
   - итоговую по продукту.
3. Если маржа стабильно в зоне риска 2+ месяца:
   - сначала корректировать подписку `S`;
   - затем, при необходимости, пересматривать `K` (редко и заранее уведомляя пользователей).

## Правила снижения рисков

- Не предоставлять безлимит AI без жестких ограничений.
- Включить проверки перед запросом:
  - достаточно ли баланса;
  - не превышены ли лимиты пользователя.
- Задать минимальный остаток для автоматического списания подписки.
- Вести журнал всех списаний на уровне запроса (id запроса, модель, `cost_rub`, `K`, `charge_rub`).

## Метрики, которые нужно смотреть

- `MRR` (выручка подписок в месяц);
- доля AI-расходов в выручке;
- валовая маржа по AI-списаниям;
- валовая маржа по продукту;
- средняя комиссия эквайринга;
- доля возвратов;
- доля неуспешных списаний подписки.

## Резюме модели

Финальная модель для запуска:

- единый фиксированный `K` без плавающих тарифов;
- ежемесячная подписка `S`;
- списание за AI строго от фактического `cost_rub` AiTunnel;
- прозрачная оферта и учет входящих платежей под НПД.

Это дает простую и понятную для пользователей систему, при этом сохраняет контроль прибыли и снижает
налоговые/операционные риски.
