# 🪄 Interactive UI-Orchestrated LLM

## ✨ Продукт: слоган и образ

**📣 Варианты слоганов** (черновики для бренда и коммуникаций):

- 🎯 «От запроса к результату — без бесконечного чата»
- ✨ «Сначала ясность — потом магия»
- 🧭 «Один замысел — понятный путь»
- 📋 «Форма вместо переписки»
- 🪜 «Скажи, что нужно — соберём детали по шагам»

**🪄 Иконка — волшебная палочка.** Визуальный символ продукта — **палочка волшебника**: она передаёт идею **превращения
намерения в готовый результат** — 📝 текст, 🖼️ картинку, 📊 диаграмму — без длинных ритуалов в чате. Это не «магия
вслепую», а **направленное действие**: палочка срабатывает после того, как сценарий собрал нужные детали (как в сказке —
✨ слова заклинания задают направление, остальное помогает дорисовать картину). Для интерфейса это задаёт образ:
**лёгкий жест — осмысленный итог** ✅

> **🏷️ Рабочее название:** «Диалог как форма» · **Prompt → UI Plan → Fill → Result**

Ниже — продуктовая концепция: как заменить бесконечные уточнения в чате на **пошаговую форму**, которую строит модель по
единому контракту (JSON), с пресетами задач и опциональным голосом.

## 📌 TL;DR

- 🧭 **Стартовый экран:** пользователь выбирает **пресет** под задачу или **«Свободный»**. Пресет — **вариация** того же
  сценария, что и свободный режим, но с **зашитыми** у нас промптами и ветками; свободный режим строит шаги и
  **финальный промпт** «на лету» (см. раздел ниже). **Цель:** для одинаковых задач свободный режим **не слабее**
  пресетов.
- 🎯 **Промпт:** короткий запрос на естественном языке; в свободном режиме допустим любой смысловой запрос.
- 🤖 Модель **не обязана** сразу выдать готовый артефакт — она **планирует экран**: что собрать и **какими виджетами**.
- 📋 Клиент строит **форму** из **машиночитаемой схемы** (поля, селекты, тумблеры), а не вытягивает данные из длинного
  текста ответа.
- ✅ Пользователь отправляет **структурированные ответы** → модель возвращает **результат** (текст, диаграмма, SVG и
  т.п.).
- 🎙️ **Голос** (см. ниже): первый промпт и ответы на шаги можно **диктовать** — без клавиатуры, тот же контракт UI Plan.

## ⚠️ Проблема

| Что не так в «чистом чате»  | Почему больно                                               |
| --------------------------- | ----------------------------------------------------------- |
| 🔄 Много раундов уточнений  | Долго, особенно на мобильном                                |
| 📝 Ответы в свободной форме | «25 лет» vs «двадцать пять» — лишняя нормализация           |
| 🎨 Нет гарантии типа ввода  | Модель «спросила число», пользователь вставил абзац         |
| ♿ Слабая доступность       | Свободный текст плохо стыкуется с масками, голосовым вводом |

---

## 💡 Идея решения

### 📌 Суть

На промежуточных шагах модель возвращает не только текст для человека, а **план интерфейса** в **универсальном формате**
— по сути **declarative UI** и **слоты для данных**. По духу это рядом с **tool calls**, но цель — **рендер формы** на
клиенте, а не вызов API в один клик.

### ➡️ Паттерн в одной строке

```text
Промпт (NL) → Схема шага (JSON) → Рендер UI → Ответы пользователя (JSON) → … → Финальный артефакт
```

## 🧭 Стартовый экран: пресеты (режимы) и свободный ввод

### ❓ Зачем стартовый экран

Перед первым вызовом оркестратора пользователь видит **каталог задач** — не пустое поле чата, а **осмысленный выбор**.
Каждый пункт — это **`modeId`** на клиенте и **дополнение к базовому системному промпту** на сервере: роль, ограничения,
типичные поля `collect`. Базовые правила (function calling, схема шагов) **общие**; пресет **не дублирует** контракт, а
задаёт **контекст и приоритеты** сценария.

### 📚 Пресеты: примерный каталог задач (расширяемый)

Ниже — ориентир для продукта; реальные `modeId` и формулировки фиксируются в коде/локалях.

| Группа                     | Примеры режимов (`modeId` ориентир)                                                                                  | Зачем людям                                                                                                                                        |
| -------------------------- | -------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **📝 Текст**               | Улучшение текста; краткое содержание; пересказ «простым языком»; черновик письма / email                             | Повседневная редактура и коммуникация                                                                                                              |
| **💌 Поздравления**        | Поздравление с уточнением адресата и повода: мама, бабушка, друг, папа, коллега, ребёнок; юбилей, Новый год, свадьба | Не придумывать структуру с нуля — форма соберёт тон, длину, обращение                                                                              |
| **💼 Идеи и бизнес**       | Бизнес-идея (проблема, ЦА, гипотезы); идеи поста / заголовки; FAQ или ответы на возражения                           | Структурированный мозговой штурм без промпт-инженерии                                                                                              |
| **📅 Выступления и планы** | Тезисы для выступления; повестка встречи; чек-лист из цели                                                           | Быстрый каркас под время и формат                                                                                                                  |
| **🎨 Визуал и вектор**     | **SVG-иконка**; **логотип**; **аватарка** (стилизованный портрет / персонаж)                                         | После генерации растра — **векторизация** (например potrace-wasm) для чистого SVG; пресеты задают монохром/контраст, стиль «иконка vs иллюстрация» |
| **📊 Данные и схемы**      | Диаграмма (столбцы, круг, линия); таблица из текста                                                                  | Сбор чисел и подписей через `collect`, выход в spec или SVG                                                                                        |

Дополнительно по вкусу продукта: «резюме вакансии», «объявление о продаже», «текст для лендинга» — тот же механизм
пресетов.

### 🆓 Пункт «Свободный» (free / custom)

Режим **без узкого сценарного пресета**: запрос формулируется как в обычном чате, но дальше работает тот же
**оркестратор с UI Plan**, а не «сырой» текст от модели.

**📋 Схема работы**

1. Пользователь выбирает **«Свободный»**.
2. На сервер уходит **`modeId: free`** (или эквивалент) и **базовый системный промпт оркестратора** — как у остальных
   режимов (`collect` / `generate`, инструменты, function calling).
3. Пользователь пишет, например: _«Создай диаграмму продаж»_ или _«Сделай иконку кота»_.
4. **Оркестратор** сам решает: нужен ли `collect`, какие поля и `kind`, нужен ли tool (картинка, векторизация,
   диаграмма). **Недостающий ввод** продумывает модель, а не заранее зашитая форма пресета.

**⚖️ Пресеты и свободный режим**

| Аспект           | **📦 Пресеты**                                                        | **🆓 Свободный**                                   |
| ---------------- | --------------------------------------------------------------------- | -------------------------------------------------- |
| Назначение       | Быстрый старт, понятные сценарии, **предсказуемый** тюнинг под задачу | **Любой** контент одной точкой входа               |
| Системный слой   | Базовый + **узкий** пресет (роль, типичные поля)                      | Только **базовый** оркестраторский слой            |
| Поведение модели | Меньше «разведки», ближе к ожидаемой форме                            | Больше рассуждения о намерении и составе `collect` |
| UX               | Ниже когнитивная нагрузка: проще понять, что спросить                 | Выше гибкость, слабее заранее заданное направление |

Оба варианта используют **один и тот же** контракт UI Plan и tools: пресет лишь **сужает** контекст и ускоряет сценарий.

## ⚖️ Пресеты и «Свободный»: одна логика, два способа её задать

### 🔁 Пресеты — вариация свободного режима, а не отдельная механика

**Пресет** — это по сути **тот же сценарий**, что и в свободном режиме (оркестратор, шаги, UI Plan, инструменты), но с
**заранее зашитыми** у нас в продукте артефактами:

- **системные промпты** под задачу и под ветки сценария;
- **что именно** модель должна спросить у пользователя (какие параметры, в каком порядке, какое поведение полей);
- при необходимости **несколько связанных промптов** — не один «монолит», а набор заготовок под выбор пользователя.

В пресете **жёстко заданы рамки** (мы уже решили, какие ветки возможны и какие вопросы уместны), но внутри рамок
пользователь по-прежнему даёт **свободные ответы** и выбирает подтипы — поэтому это **узкий коридор**, а не тупик.

### 🧩 Как пресет собирает финальный промпт (пример: диаграммы)

Пользователь идёт по пресету «диаграмма» и на шаге выбирает, например, **круговую** диаграмму или **3D**-чарт. Мы **не**
полагаемся на то, что модель «на память» знает лучший способ описать круговую vs 3D: у нас **сохранены** отдельные
промпт-куски (или целые шаблоны) под каждый подтип.

**Финальный промпт** в генерацию собирается **у нас**: из ответов пользователя + **подставляется** сохранённый промпт
выбранной ветки (круговая / 3D / столбцы и т.д.). То есть «магия формулировки» под визуальный тип **зашита в продукте**;
модель в этом месте исполняет предсказуемую роль.

Итог: пресет **достаточно жёстко ограничен** (мы контролируем ветвление и шаблоны), но **свобода пользователя** остаётся
в данных, формулировках и выборе подтипа.

### ✨ Свободный режим: то же поведение — «на лету»

В **свободном** режиме те же вещи (какие параметры спросить, как собрать **финальный промпт** под выбранный тип
результата) должны возникать **динамически**: мы даём модели **инструкции оркестратора**, и она **на лету** планирует
шаги `collect`, поля и вызовы инструментов — без заранее сохранённого дерева пресета.

Дополнительно заложена **двухфазная** логика подачи работы в модель генерации результата:

1. По нашим **мета-промптам** (инструкции вида: «сгенерируй финальный промпт для следующего шага исходя из намерения
   пользователя и уже собранных ответов») модель **сначала выдаёт финальный промпт** — явно сформулированный текст (или
   структуру), готовый к исполнению.
2. Затем этот **финальный промпт обрабатывается** — передаётся в следующий этап (генерация картинки, диаграммы, текста и
   т.п.) так, как задумано оркестратором.

Так свободный режим **компенсирует** отсутствие заранее сохранённых веток: часть «как в пресете» переносится в этап
**сборки финального промпта** по правилам, которые мы задаём мета-инструкциями.

### 🎯 Цель паритета качества

**Продуктовая цель:** для **одних и тех же задач** (та же диаграмма, тот же тип иконки, тот же сценарий поздравления)
поведение в **свободном** режиме должно быть **не хуже**, чем в режиме **пресетов** — по полноте вопросов, по качеству
итогового артефакта и по предсказуемости шагов.

Пресеты задают **планку** и эталон «как мы хотим, чтобы это работало при известном сценарии». Свободный режим не должен
восприниматься как **упрощённая** или **более слабая** версия для тех же целей: он лишь **другой способ** дойти до
схожего качества — через **динамическое** планирование и явную фазу **генерации финального промпта** перед исполнением.

## 📖 Ключевые понятия

| Термин                 | Описание                                                                                              |
| ---------------------- | ----------------------------------------------------------------------------------------------------- |
| 📐 **UI Plan**         | Структура экрана: поля, типы, обязательность, подсказки                                               |
| 🏷️ **Field kind**      | Тип виджета: `text`, `number`, `select`, `toggle`, …                                                  |
| 🔢 **Step**            | Фаза диалога: сбор данных, генерация, подтверждение                                                   |
| 📦 **Artifact**        | Промежуточный продукт: кандидаты палитр, черновики                                                    |
| ✅ **Collected state** | Накопленные ответы пользователя по `id` полей                                                         |
| 🎙️ **Voice adapter**   | Слой «речь ↔ значения полей»: ASR, маппинг на `kind`, опционально TTS                                 |
| 🔧 **Tool**            | Именованное действие после `collect`: генерация изображения, векторизация, отрисовка чарта и т.п.     |
| 🎛️ **Mode preset**     | Идентификатор задачи со стартового экрана → дополнение к системному промпту без смены контракта формы |
| ♾️ **Free mode**       | Тот же оркестратор, но без узкого пресета: модель сама предлагает шаги и поля под произвольный запрос |

## 🏗️ Архитектура: универсальный язык (контракт)

Ответ модели — **валидируемый JSON**; на клиенте желательны **JSON Schema** и при необходимости проход **repair**
(исправление до соответствия схеме).

### 📄 Минимальная форма шага

```json
{
  "step": "collect",
  "title": "Данные для графика",
  "message": "Заполните поля ниже — так я смогу построить чарт без ошибок.",
  "fields": [
    {
      "id": "chart_title",
      "label": "Заголовок",
      "kind": "text",
      "required": true,
      "placeholder": "Например: Продажи по кварталам"
    },
    {
      "id": "x_labels",
      "label": "Подписи по оси X (через запятую)",
      "kind": "textarea",
      "required": true
    },
    {
      "id": "values",
      "label": "Значения (через запятую)",
      "kind": "textarea",
      "required": true
    },
    {
      "id": "palette",
      "label": "Палитра",
      "kind": "select",
      "required": true,
      "optionsFrom": "palette_candidates"
    }
  ],
  "artifacts": {
    "palette_candidates": [
      { "id": "p1", "label": "Ocean", "colors": ["#0ea5e9", "#0369a1", "#164e63"] },
      { "id": "p2", "label": "Sunset", "colors": ["#f97316", "#ea580c", "#9a3412"] }
    ]
  }
}
```

### 🏁 Финальный шаг (пример)

```json
{
  "step": "generate",
  "result": {
    "type": "chart_spec",
    "format": "vega-lite",
    "payload": {}
  }
}
```

### 🧾 Whitelist типов полей (`kind`)

Рекомендуется **жёсткий список** на клиенте — модель не может «придумать» неподдерживаемый виджет.

| kind           | Виджет            | Примечание                          |
| -------------- | ----------------- | ----------------------------------- |
| `text`         | Однострочный ввод | Имена, короткие строки              |
| `textarea`     | Многострочный     | Списки через запятую, абзацы        |
| `number`       | Число             | min/max в `constraints`             |
| `select`       | Один вариант      | `options` или `optionsFrom`         |
| `multiselect`  | Несколько         | Теги, стили форматирования          |
| `toggle`       | Да/Нет            | Emoji, юмор, «согласен с условиями» |
| `color`        | Цвет              | Один key color или акцент           |
| `file`         | Файл              | Референсы, CSV                      |
| `chart_series` | Составной блок    | Строки × колонки (табличный ввод)   |

Расширяйте список только осознанно — каждый новый `kind` = работа UI и тестов.

## 🌊 Потоки (примеры)

### 📊 График (chart)

1. 👤 Промпт: _«Столбчатый график выручки по кварталам»_.
2. 🤖 `collect`: заголовок, подписи оси X, значения, тип чарта (`select`), опционально единицы измерения.
3. 🎨 Модель (или отдельный лёгкий вызов) формирует **`palette_candidates`** → клиент показывает **select** с превью
   цветов.
4. ✅ Отправка ответов → `generate` → **`chart_spec`** (Vega-Lite, ECharts JSON) или **SVG** — клиент только рендерит.

### 💌 Поздравление (greeting)

1. 👤 Промпт: _«Сгенерируй поздравление для друга»_.
2. 🤖 `collect`:
   - возраст → `number`;
   - имя → `text`;
   - стиль → `select` (шутливый / нейтральный / деловой / трогательный);
   - emoji → `toggle`;
   - длина → `select` (коротко / средне / развёрнуто);
   - форматирование → `multiselect` (абзацы, список, заголовок).
3. ✅ `generate` → Markdown с учётом тумблеров и стиля.

### 🧩 Общий принцип

- 🔁 Шагов может быть **несколько**: `collect` → `collect` (уточнение) → `confirm` → `generate`.
- 🧠 Состояние — это **объект ответов** по `id`, плюс **история промптов** при необходимости.

## 🖼️ Генератор изображений и инструменты (частный случай)

Тот же **UI Plan** и цепочка `collect` → `generate`. На выходе модель может указать **вызов инструментов** — по смыслу
как **tool / function calling**, но запуск не из свободного чата, а **после заполнения формы**.

**Продукт (контекст):** генератор с **фиксированным набором настроек** — стиль, палитра, фотореализм, назначение
(**иконка** vs **изображение**). Пользователь даёт **короткое описание** и настраивает слайдеры и селекты вместо
длинного ручного промпта; такой UX рассчитан на широкую аудиторию.

**Диалог с текстовой моделью:** на `collect` — варианты стиля, назначения, **кандидаты палитр** в `artifacts` или ввод
своей палитры; поля и `kind` те же, что в остальных сценариях. На `generate` текстовая модель отдаёт **структурированный
промпт и параметры** для **модели изображений**; та возвращает растр.

**Инструменты по необходимости:**

- **Растровая картинка** — базовый `generate_image` (или эквивалент) по согласованным полям.
- **SVG-иконка** — при выборе назначения «иконка» / запросе вектора: сначала генерация **монохромного или контрастного**
  растра под векторизацию, затем **векторизация в браузере**
  ([potrace-wasm](https://github.com/IguteChung/potrace-wasm)) → SVG; это отдельный **tool** в цепочке после растра.
- **3D-диаграммы** (bar, pie и т.д. с подписями) — тот же сбор **данных и палитры**, иной **`result`** или tool-вызов
  рендера 3D-чарта; не смешивать с покадровой генерацией «как фото», если нужна читаемая диаграмма.

Интеграция в план: в JSON шага `generate` достаточно полей вроде `tools: [{ "name": "...", "args": { } }]` и/или
`result.type` с известными клиенту значениями — клиент и оркестратор **маршрутизируют** на генератор изображений,
векторизатор или движок диаграмм без дублирования логики формы.

---

## 🎙️ Голосовой режим: промпт и форма без клавиатуры

Идея допускает полностью **голосовой** сценарий поверх того же контракта UI Plan: клавиатура не обязательна, если клиент
умеет превращать речь в **текст** и сопоставлять его с **активным полем** и **допустимыми ответами**.

### 👤 Как это выглядит для пользователя

- 🎤 **Старт:** пользователь **диктует** первый промпт («сделай график продаж», «поздравь друга»). Аудио идёт в
  **транскрибацию (ASR)** → в модель уходит уже **текст** (или текст + метаданные сессии).
- 🔊 **Микрофон на время сессии голосового режима:** после ответа модели пользователь **не уходит в клавиатуру** —
  **голосом** отвечает на вопросы сценария: числа, имена, выбор варианта («первый», «второй», «нейтральный стиль»),
  да/нет для тумблеров.
- 📋 **Форма остаётся той же:** клиент по-прежнему рендерит поля из JSON, но **фокус** и **подсказки** заточены под
  голос: что сказать, какие варианты пронумерованы для выбора речью.
- 🔁 **Цикл:** речь → ASR → нормализация под `kind` поля → запись в `answers` → при необходимости новый запрос к модели
  → **TTS** озвучивает следующий вопрос или итог (по желанию продукта).

### 🔗 Связка «речь → структура»

| Задача                    | Поведение                                                                                                                          |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| 📝 Свободный текст в поле | Прямая подстановка результата ASR (с лёгкой очисткой пробелов и пунктуации).                                                       |
| 🔢 Число                  | Распознавание цифр и слов («двадцать пять» → `25`); при неуверенности — уточняющая реплика или повтор.                             |
| 📌 `select`               | Пользователь называет **лейбл**, **синоним** или **номер** варианта («второй», «как Ocean»); клиент маппит на `option.id`.         |
| ✅ `toggle`               | Явные фразы: «да / нет», «включи / выключи», «с эмодзи / без».                                                                     |
| 🎨 Палитры                | «Первый набор цветов», «Sunset», «тот что с оранжевым» — сопоставление с `palette_candidates` через LLM или правила + fuzzy match. |

При необходимости **один короткий вызов LLM** только на этапе маппинга: _«вот варианты select, вот что сказал
пользователь — верни id или null»_ — чтобы не гонять полный UI Plan в цикле.

### 🔧 Технические заметки (без привязки к вендору)

- 🎧 **ASR:** потоковая транскрибация предпочтительна при «всегда слушаем» — чтобы уменьшить задержку; VAD (voice
  activity detection), чтобы не слать тишину.
- 🔈 **TTS:** для доступности и «hands-free»; громкость/наушники — чтобы колонки не создавали ложные срабатывания
  микрофона (AEC — акустическое эхоподавление на клиенте, если доступно).
- 🔒 **Приватность:** явный индикатор записи, политика хранения аудио (часто хранят только текст), опция «голос только
  на устройстве» если появится локальная ASR.

### 🤝 Зачем это сочетается с UI Plan

Тот же JSON с `fields` — **единый источник правды**: и **тап по экрану**, и **голос** заполняют один объект `answers`.
Режим ввода — свойство клиента, а не модели. Это упрощает тесты: одни и те же схемы, разные input-адаптеры (**keyboard**
vs **voice**).

## 📈 Преимущества перед «модель спрашивает в чате»

- ⚡ Меньше сообщений туда-сюда.
- 🎯 Типизированный ввод — проще валидация и автоподстановки.
- 🧪 Проще автотесты: схема + фикстуры ответов пользователя.
- ♿ Предсказуемая разметка — лучше для a11y и мобильных клавиатур.

## 🛡️ Риски и смягчение

| Риск                              | Что делать                                                                                                  |
| --------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| 📉 Невалидный JSON от модели      | JSON Schema + retry / «исправь до соответствия схеме»                                                       |
| 🔢 Слишком много полей            | Лимит в системном промпте; разбивка на **шаги**                                                             |
| 🎨 Галлюцинации палитр            | Детерминированный генератор палитр от темы + seed; или отдельный короткий вызов                             |
| 🧱 Расширение `kind` без контроля | Только whitelist; неизвестное поле → отбраковка или безопасный fallback                                     |
| 🔐 Ввод персональных данных       | Политика хранения; не логировать лишнее; опционально локальная обработка                                    |
| 🎙️ Ошибки ASR / шум               | Подтверждение распознанного текста на экране; повтор фразы; хороший VAD; подсказки «скажите номер варианта» |
| 🔁 Конфликт TTS и микрофона       | Приоритет TTS, mute mic на время озвучки или AEC; короткие реплики                                          |
| 👂 «Всегда включённый» микрофон   | Прозрачный UI, отзыв согласия, чёткая граница сессии записи                                                 |

## 🚀 MVP (минимальный скоуп)

1. 🧭 Стартовый экран: **несколько пресетов** + пункт **«Свободный»**; передача `modeId` в каждый шаг API.
2. 📝 Системный промпт: ответ **только JSON** по заданной схеме; шаги `collect` | `generate`.
3. 🖥️ Клиент: динамический рендер полей по `fields[]`.
4. 💾 Состояние сессии: объект `answers`, merge после каждого сабмита.
5. ✔️ Валидация обязательных полей на клиенте до отправки.
6. 🔄 Сценарии end-to-end: например **поздравление** (пресет) и **свободный запрос** → диаграмма или текст.
7. 🎙️ (Фаза 2) Голос: ASR → заполнение `answers` + TTS для вопросов; сценарии **select** с пронумерованными вариантами.

- **«Уточняющий режим»** — мягко, понятно не технарям.

## 🔗 Связанные идеи и аналоги

- 🛠️ **Function / tool calling** — вызов функций; **рендер формы** остаётся основой, **tools** исполняются **после**
  заполнения слотов (см. «Генератор изображений и инструменты»).
- 📋 **JSON Forms** / **RJSF** — похожий принцип: схема → UI.
- 🧙 **Multi-step wizard** — тот же UX, но план шагов задаёт **LLM** под запрос пользователя.
- 🗣️ **Голосовые ассистенты (voice UIs)** — тот же принцип слотов и подтверждений; здесь слоты формализованы **UI Plan**
  и общие для голоса и тапов.

## 🎯 Итог

Идея — **совместить свободу первого промпта** с **дисциплиной формы** в середине диалога: модель — **оркестратор
интерфейса**, клиент — **исполнитель** контракта.

**Стартовый экран с пресетами** снижает порог входа и задаёт контекст под типичные задачи (текст, поздравления,
бизнес-идея, иконки, логотипы, аватарки, диаграммы). **«Свободный»** использует тот же механизм, но шаги и поля модель
проектирует под произвольный запрос. Дальше сценарий масштабируется от коротких поздравлений до сложных диаграмм и
экспорта артефактов.

**Голос** контракт не меняет: UI Plan заполняется через **ASR**, при желании шаги озвучивает **TTS**. **Картинки,
векторизация, 3D-диаграммы** — тот же цикл: **слоты и настройки**, затем **маршрут на нужный tool**.
